### Chitta delete guard

The `delete_memory` tool does a hard delete. This is risky when other agents (subagents, routines) have access to chitta. Plan: make `delete_memory` a soft-delete that sets `invalidated_at` timestamp instead. Only a separate purge operation (or direct psql) should hard-delete. Search already has the `invalidated_at` column in the ogham schema but chitta-rs doesn't use it yet. Wire it up when doing the next chitta-rs update.

Interim mitigation: add a PreToolUse hook gating `mcp__chittars__delete_memory` (checks `$CLAUDE_CODE_AGENT_DEPTH`, blocks subagents).

## BGE-M3 considerations (2026-04-27 chat)

Conversational notes on whether known embedding-space gotchas affect chitta. Not a plan, just things to keep in mind when deciding what to test next.

### Hybrid retrieval — already tested, mixed picture

PersonaMem retrieval-only sweep (`bench/results/pre-v0003-tests/rrf-sweep-personamem/results-overview.md`) showed **sparse adds +2.9pp recall@20 over dense-only (93.9% → 96.8%)** with no latency cost. FTS adds nothing on top. rrf_k barely matters. The overview's own recommendation was to enable `rrf_sparse` as production default.

But end-to-end RAG accuracy on the same dataset was 60–64% — the ~30pp gap to retrieval recall says **the answer LLM, not retrieval, is the bottleneck**. So sparse "helps" in the narrow recall sense but doesn't move the needle on what users feel.

LifeBench and BEAM rrf-sweeps are in `bench/results/pre-v0003-tests/{lifebench,beam}-rrf-sweep/` — BEAM run was reportedly corrupted; LifeBench has raw retrieval JSONs but no summary md was written. Worth a follow-up: write the lifebench summary, redo beam, then decide if "no improvement" is true everywhere or just on RAG-end-to-end.

Open question worth answering before re-litigating sparse: **does sparse help on technical/identifier-heavy queries** (file paths, symbol names, error strings, UUIDs)? PersonaMem and LifeBench are natural-language-ish. Chitta's actual workload includes a lot of `qartez_impact`, `.sessions/<uuid>.jsonl`, commit SHAs — the kind of tokens dense embeddings blur and sparse pins. None of the existing benches exercise that distribution. A small synthetic eval ("find the memory mentioning `<exact identifier>`") would settle whether sparse is worth the v0.0.3 wiring for chitta's *own* use case even if it didn't move PersonaMem RAG.

### Other gotchas, ranked by likely relevance

1. **What gets embedded.** If the embedded text is just `content`, tags like `decision` / `session-summary` / `qartez` aren't contributing retrieval signal beyond filterability. Worth confirming whether tags are concatenated into the embedded string. If not, that's a cheap recall win.

2. **Query instructions.** BGE-M3 was trained with task-prefix instructions (`"Represent this sentence for searching relevant passages: "` on the query side, nothing on the doc side). Skipping the prefix can cost a few points of recall. Worth checking how the embedder leg constructs the query input vs the doc input.

3. **Hubness.** Real in any high-dim retrieval system but usually only matters at the top-K extremes. Symptom: the same memory keeps surfacing for unrelated queries. Worth a sanity script that for each memory counts how often it appears in top-K across a random query set — outliers are hubs.

4. **Anisotropy.** Less of a concern. Contrastively-trained retrieval models like BGE-M3 spread vectors better than vanilla LMs. Absolute scores will still cluster (most pairs in 0.4–0.85), but ranking quality is what matters.

5. **Hardcoded similarity thresholds.** If anywhere in chitta-rs filtering uses `score > 0.7` or similar absolute cutoffs, those are arbitrary across models/datasets. Better: top-K + relative gap, or a re-ranker. Worth a grep for absolute thresholds in `src/retrieval/`.

### Suggested order if revisiting this

1. Write the LifeBench rrf-sweep summary from existing JSONs (no rerun).
2. Grep chitta-rs for: (a) what fields go into the embedded string, (b) whether query input is prefixed, (c) any hardcoded score thresholds.
3. Build a tiny identifier-recall eval against actual chitta memories.
4. *Then* decide if hybrid retrieval is worth the engineering for chitta itself, separately from what PersonaMem said.
