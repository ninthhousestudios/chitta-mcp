### Chitta delete guard

The `delete_memory` tool does a hard delete. This is risky when other agents (subagents, routines) have access to chitta. Plan: make `delete_memory` a soft-delete that sets `invalidated_at` timestamp instead. Only a separate purge operation (or direct psql) should hard-delete. Search already has the `invalidated_at` column in the ogham schema but chitta-rs doesn't use it yet. Wire it up when doing the next chitta-rs update.

Interim mitigation: add a PreToolUse hook gating `mcp__chittars__delete_memory` (checks `$CLAUDE_CODE_AGENT_DEPTH`, blocks subagents).
