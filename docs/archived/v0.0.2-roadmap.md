# chitta-rs v0.0.2 roadmap

v0.0.2 makes chitta-rs usable as a daily driver alongside Python chitta. New memories go here; migration from Python chitta is deferred to v0.0.3.

---

## Phase 0 — Research foundations

Retrieval quality research is a primary goal of chitta-rs, not an afterthought. These architectural properties ship in v0.0.2 so that every query from day one generates usable evaluation data.

1. **Query log table.** Every `search_memories` call logs: query text, embedding vector, parameters (k, min_similarity, tags), result IDs with scores, latency_ms, timestamp. Append-only, no impact on response path. This is the replay corpus for comparing retrieval strategies.
2. **Pluggable retrieval pipeline.** The search path should be a composable pipeline (embed → retrieve → score → rank → truncate) where stages can be swapped or inserted without rewriting the tool layer. Even if v0.0.2 only ships one strategy (pure cosine similarity), the architecture must allow slotting in BM25, rerankers, or hybrid RRF as additional stages.
3. **Baseline metrics helper.** A CLI subcommand or script that replays logged queries against the current retrieval config and outputs precision/recall/MRR scores. Doesn't need to be fancy — just reproducible. This is how you know whether a change helped.

---

## Phase 1 — Embedder hardening

Fix the concurrency and reliability issues that block HTTP transport and affect stdio robustness.

1. **Embedder session pool.** Replace global `Mutex<Session>` with a pool of N sessions (N = num_cpus or configurable) behind a `Semaphore`. Concurrent requests embed in parallel instead of serializing.
2. **Mutex poison recovery.** A panic in `session.run()` currently kills the server permanently. Wrap in `catch_unwind` inside `spawn_blocking`, or use the session pool to discard the poisoned session and create a fresh one.
3. **Skip embed on idempotent replay.** Pre-flight `SELECT` on `(profile, idempotency_key)` before embedding. Replays skip the full ONNX cost.
4. **Token-length guard validation.** At `Embedder::load` time, check whether `tokenizer.json` has truncation enabled. If so, warn or disable it so the `MAX_TOKENS` check actually fires.

## Phase 2 — New tools

The minimum tool surface for daily use beyond store/get/search.

5. **`update_memory`** — Update content and/or tags of an existing memory. Re-embeds if content changes.
6. **`delete_memory`** — Hard delete by profile + id.
7. **`list_recent_memories`** — Time-ordered retrieval (most recent first). Supports limit and optional tag filter. Semantic search doesn't cover "what happened recently."

## Phase 3 — Streamable HTTP transport

One persistent process, multiple Claude Code sessions.

8. **`--http` flag.** Streamable HTTP (MCP 2025-11-05 spec) via rmcp's `StreamableHttpServerConfig`. Same binary, same tools, different transport.
9. **Claude Code config shape:** `{ "type": "http", "url": "http://localhost:PORT/mcp" }`.
10. **Bearer-token auth.** `--auth-token-file` flag. Required for `--http`, rejected for stdio.
11. **Graceful shutdown.** HTTP mode needs clean connection draining on SIGTERM.

## Phase 4 — Search correctness

Fix known bugs that cause incorrect agent behavior.

12. **`truncated` flag false positive.** `total_available` ignores `min_similarity`, causing spurious `truncated=true`. Agents may pagination-loop. Fix: count query should apply the same `min_similarity` filter.
13. **Count query inside transaction.** Move the count query inside the search transaction to avoid MVCC skew under concurrent writes.
14. **Tags filter test coverage.** The SQL branch for tag filtering exists but is untested. Add integration tests with non-empty tag arrays.

## Phase 5 — Contract alignment

Decide and document the contract story before v0.0.3 migration.

15. **Audit Python vs Rust contract differences.** Document every breaking change: field renames (`memory_id` -> `id`, `limit` -> `k`, `score` -> `similarity`), required fields (`profile`, `idempotency_key`), dropped fields (`source`, `metadata`, `auto_link`).
16. **Decision: converge or diverge.** Either chitta-rs adopts Python's field names (easier migration, familiar to existing CLAUDE.md instructions) or stays opinionated and the cutover updates all call sites. Decide before v0.0.3.

---

## Out of scope (v0.0.3+)

- **Memory migration from Python chitta.** Bulk import tooling, schema mapping, embedding re-generation or transfer. This is v0.0.3's headline.
- Hybrid search (BM25 + semantic RRF).
- Memory graph, relationships, contradictions.
- SimHash dedup, compression, importance/confidence scoring.
- Profile management tools, TTL, expiry.
- `metadata jsonb` column.
